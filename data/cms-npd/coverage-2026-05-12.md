# Fleet identity coverage — 2026-05-12

Per-vendor coverage of authoritative identifiers attached to each FHIR
endpoint in `ehrs/{vendor}/production_fleet.json`. Sources are layered:

  1. **NPD** — CMS National Provider Directory (NDH-style). Endpoint URL
     joined to `Organization` (NPI), parent system via
     `OrganizationAffiliation`.
  2. **POS (deterministic)** — CMS Provider-of-Services file. Now covers
     all FHIR-publishing-eligible facility categories (Hospital, FQHC,
     ASC, RHC, SNF, ESRD, …) — not just hospitals. Source of CCN via
     name+address Jaccard against brands-bundle Organization addresses.
  3. **NPPES** — CMS National Plan & Provider Enumeration System. Full
     org name lookup. Includes URL exact-match against NPPES's own
     endpoint registry plus `rapidfuzz` token-set name match within
     state. Source of NPI fallback and NUCC taxonomy.
  4. **POS via NPPES (Tier-3 augmentation)** — for fleet endpoints with
     an NPPES-resolved NPI but no CCN from layer 2, uses the NPPES
     practice address as the join key against POS. Picks up the
     ambulatory / patient-portal endpoints whose brands-bundle
     Organizations had empty addresses.
  5. **LLM disambiguation (with NUCC pre-filter)** — Claude Haiku 4.5
     picks the best CCN from `candidates[]` for multi-candidate POS
     rows the deterministic resolver couldn't auto-accept. Candidates
     are pre-filtered by `tools.nucc_to_pos`: an endpoint whose NPPES
     taxonomy says "Acute Care Hospital" only sees hospital-category
     POS candidates, suppressing FQHC/ASC noise. High+medium confidence
     picks become overrides; low/none stays out. Costs ~$1.50 per full
     run; cached per (endpoint, candidate-set, model) so re-runs are
     free. Run `tools.recover_llm_overrides --all` after re-running the
     LLM tool to rebuild the override file from the full cumulative
     cache rather than just the most-recent increment (the LLM tool
     overwrites in place; the recovery tool unions across runs).
  6. **PECOS** — CMS Public Provider Enrollment File. Adds PAC ID (stable
     across NPI changes — longitudinal join key) and CMS-canonical
     Medicare provider-type description.

## What each source contributes uniquely

- **NPD**: authoritative endpoint→org linkage for the ~16% of orgs that
  registered themselves into NDH. Strongest signal because the
  registration is explicit.
- **POS**: CCN for Medicare-certified facilities. Broader coverage in
  this release (53k providers across 16 categories vs prior 7k hospitals).
  Cerner/Epic ambulatory deployments benefit most from the FQHC/ASC/RHC
  rollup.
- **NPPES**: the 80× expansion. Two strategies do most of the work:
  - URL exact match against NPPES's own `endpoint_pfile` (46k FHIR/REST
    endpoints with direct NPI attribution).
  - Fuzzy name match within state (`rapidfuzz` token-set ratio ≥85)
    catches name variants like "RWJ Barnabas Health" → "ROBERT WOOD
    JOHNSON BARNABAS HEALTH" that exact normalization misses.
- **PECOS**: PAC ID + provider-type description for the subset of
  resolved NPIs that are actively enrolled in Medicare. PAC ID stays
  stable across NPI changes (merger/sale/revocation), so it's the most
  durable longitudinal key.

## Known ceilings and limits

- **Shared-tenant hosts** (`fhir-myrecord.cerner.com`,
  `epicproxy-pub.et*.epichosted.com`) defeat URL-and-hostname strategies
  by definition: every path is a different tenant. Only name-based
  matching helps for these.
- **Infrastructure-jargon names** like "MaaS MEDITECH Hosted Pod 009",
  "CPHY Production Central Time Zone B" are filtered out — they're
  vendor sandbox/tenancy labels, not real orgs.
- **MEDITECH cap at 86% NPI**: the residual 14% are on-prem deployments
  whose brand-bundle labels carry MEDITECH-internal jargon ("MaaS Pod
  ###") rather than the underlying hospital's name. Resolving them
  would require a vendor-side tenant→hospital dictionary that MEDITECH
  has not published.
- **Cerner cap at 67% NPI**: the residual ~1,000 endpoints are largely
  the patient-portal cluster on shared infrastructure, where brand-bundle
  Organization names point to sandbox or test customers without state
  hints. Tier-3 enhancement opportunity: feed NPPES-resolved addresses
  back into POS resolution for these endpoints.

## How to reproduce

```bash
# One-time index builds (output files are gitignored — regenerable):
python -m tools.fetch_cms_npd                         # download NPD bulk
python -m tools.build_npd_endpoint_index --release 2026-05-07
python -m tools.build_pos_hospital_index --categories all  # 53k facilities
python -m tools.build_nppes_index                     # latest monthly zip
python -m tools.build_pecos_index                     # latest quarterly PPEF

# Run in this order — each step depends on prior outputs:
python -m tools.resolve_endpoints_to_npd --all        # NPI from NDH
python -m tools.resolve_endpoints_to_pos --all        # CCN deterministic + LLM
python -m tools.resolve_endpoints_to_nppes --all      # NPI fallback + fuzzy
python -m tools.analyze_fleet_drift epic              # first pass — populates NPI
python -m tools.analyze_fleet_drift cerner
python -m tools.analyze_fleet_drift meditech
python -m tools.augment_pos_via_nppes --all           # Tier-3: NPPES addrs → POS
python -m tools.llm_disambiguate cerner               # optional, costs ~$0.50
python -m tools.llm_disambiguate meditech             # optional, costs ~$0.15
python -m tools.llm_disambiguate epic                 # optional, mostly cached
python -m tools.resolve_endpoints_to_pos --all        # re-apply LLM overrides
python -m tools.analyze_fleet_drift epic              # final pass
python -m tools.analyze_fleet_drift cerner
python -m tools.analyze_fleet_drift meditech
```

## Known sharp edges

- **POS catalog breadth × Epic LLM**: the broader `--categories all` POS
  catalog adds FQHCs, ASCs, RHCs, etc. to LLM candidate lists. For Epic
  (almost all real hospitals) this increases ambiguity — the LLM
  correctly returns more `none` verdicts than it would against a
  hospital-only catalog, costing Epic CCN coverage. The committed
  `data/hospital-overrides/epic-pos.llm.json` was generated with the
  hospital-only catalog and is preserved. Future improvement: pre-filter
  candidates by NPPES taxonomy before sending to LLM so Epic's clinical
  category (Acute Care Hospital, mostly) narrows the candidate pool.
- **Re-augment order**: `augment_pos_via_nppes` must run AFTER first
  `analyze_fleet_drift` (so NPPES NPIs are in the fleet) and BEFORE
  final `analyze_fleet_drift` (so the augmented overlay is loaded). The
  tool skips endpoints already in the `{vendor}-pos.json` overlay to
  avoid re-resolving what the deterministic+LLM pipeline already picked.
